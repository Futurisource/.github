# Contributing to Futurisource Labs 🛠️

Welcome! We're thrilled you want to contribute. This guide covers everything you need to go from zero to merged PR.

---

## 🌱 First Time Here?

1. **Join our Discord** → [discord.gg/futurisourcelabs](https://discord.gg/futurisourcelabs)
2. Introduce yourself in `#introductions`
3. Browse open issues with the `good first issue` label
4. Comment on an issue to claim it — no silent PRs please!

---

## 🗺️ Contribution Pathways

Everyone starts somewhere. Here's how you grow at Futurisource Labs:

### 🔰 Newcomer
Your first steps:
- Fix a bug labeled `good first issue`
- Improve documentation or a README
- Write or fix tests
- Report a bug with a clear reproduction

### 🔧 Contributor
Once you've merged 2+ PRs:
- Build a feature from an open issue
- Review other contributors' PRs
- Help answer questions in Discord `#help`

### 🏗️ Builder
Once you've merged 10+ PRs or led a feature:
- Own a sub-module or feature area
- Propose new tools via Tool Proposal
- Run a Tool Jam session

### 🔑 Core Member
Nominated by the community:
- Drive roadmap decisions
- Mentor newcomers
- Manage releases

### 🧠 Maintainer
Org-level ownership:
- Final review on critical PRs
- Org-wide architectural decisions
- Represent Futurisource Labs publicly

## 📋 How to Contribute Code

### 1. Fork & Clone
```bash
git clone https://github.com/futurisource-labs/<repo-name>.git
cd <repo-name>
```

### 2. Create a Branch
```bash
git checkout -b feat/your-feature-name
# or
git checkout -b fix/the-bug-you-are-fixing
```

### 3. Make Your Changes
- Keep commits small and focused
- Write clear commit messages: `feat: add file watcher utility`
- Add or update tests if applicable
- Update docs if your change affects usage

### 4. Open a Pull Request
- Use the PR template provided
- Link the issue it closes: `Closes #123`
- Add a short description of what changed and why
- Tag `@futurisource-labs/reviewers` for review

## 💡 Proposing a New Tool

Got an idea for a tool? Amazing.

1. Open a **New Tool Proposal** issue using the template
2. Describe: what the tool does, who it helps, rough tech stack
3. Community votes with 👍 / 👎 over 7 days
4. If approved, a repo is created using the `project-template`
5. The proposer gets first dibs to lead the build

## 🏷️ Issue Labels

| Label | Meaning |
|---|---|
| `good first issue` | Great for newcomers |
| `help wanted` | We need someone to own this |
| `bug` | Something is broken |
| `enhancement` | New feature or improvement |
| `idea` | Proposed tool — open for voting |
| `docs` | Documentation only |
| `blocked` | Waiting on something external |

## ✅ Pull Request Checklist

Before submitting, confirm:
- [ ] Code runs without errors
- [ ] Tests pass locally
- [ ] Docs updated if needed
- [ ] Follows the project's code style
- [ ] PR description explains the "why"
- [ ] Linked to an open issue

## 🙏 Recognition

Every merged PR gets:
- Your name in the `CONTRIBUTORS.md` file
- A shoutout in `#changelog` on Discord
- Progress toward your next contributor level

Monthly top contributors get a **Spotlight post** on our blog.
